# Day6：Bug 报告（Bug Report）

---

# 一、什么是 Bug 报告

Bug 报告：

**测试人员发现缺陷后提交给开发人员的报告。**

作用：

**帮助开发人员复现并修复 Bug。**

---

# 二、Bug 报告包含哪些信息

一个完整 Bug 报告通常包含：

| 字段     | 说明       |
|--------|----------|
| Bug ID | 缺陷编号     |
| 标题     | Bug 简要描述 |
| 环境     | 系统环境     |
| 复现步骤   | 操作步骤     |
| 预期结果   | 正确结果     |
| 实际结果   | 实际表现     |
| 严重程度   | Bug影响程度  |
| 优先级    | 修复紧急程度   |

---

# 三、好的 Bug 标题

Bug 标题要：

- 简洁  
- 准确  
- 包含关键信息

---

坏例子：

```
登录有问题
```

---

好例子：

```
登录页面点击登录按钮无响应
```

---

# 四、Bug 复现步骤

复现步骤要：

- 清晰  
- 完整  
- 可重复

---

示例：

```
1 打开登录页面
2 输入账号：admin
3 输入密码：admin123
4 点击登录按钮
```

---

# 五、Bug 示例

Bug 标题：

注册页面密码小于6位仍然注册成功

---

复现步骤：

```
1 打开注册页面
2 输入用户名 test01
3 输入密码 123
4 点击注册
```

---

预期结果：

注册失败，并提示密码长度不足。

---

实际结果：

系统注册成功。

---

严重程度：

High

优先级：

High

---

# 今日重点总结

Bug 报告必须：

清晰  
可复现  
描述准确

Bug 标题和复现步骤非常重要
